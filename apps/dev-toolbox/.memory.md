## dev-toolbox

Selectable Node.js, Python, Rust, JVM, and project tooling. It exists separately from
`shell-toolbox` (host-only shell utilities) and `jetbrains-toolbox` (GUI IDE manager)
so general-purpose runtimes do not inherit either app's lifecycle.

## Implementation

- Uses `docker.io/ubuntu:24.04` as a runtime/download environment with native
  build dependencies (`build-essential`, `pkg-config`, `libssl-dev`, `cmake`,
  `clang`) for Rust crates and Node native modules.
- The `.packages` wizard offers `node`, `uv`, `firecrawl`, `rust`, `sdkman`,
  `rust-lsp`, `java-lsp`, and `kotlin-lsp`. Uninstalled tools start unchecked;
  installed tools are detected from their shared-home payload paths.
- `dev-toolbox-install --tools "..."` installs selected payloads under
  `~/.local/share/dev-toolbox` and removes deselected payloads. Node defaults to
  major 22 and can be overridden with `--node-major` or
  `DEV_TOOLBOX_NODE_MAJOR`. Firecrawl is installed from the `firecrawl-cli` npm
  package into its own prefix and automatically selects the managed Node runtime.
  Rust intentionally keeps the standard `~/.cargo` and `~/.rustup` homes to
  preserve existing toolchains and cargo-installed tools from the previous
  `jetbrains-toolbox` behavior. The installer adds rustup stable if missing
  and installs `bacon` if missing. `rust-lsp` automatically selects `rust` and
  installs the `rust-analyzer` component through rustup.
  SDKMAN intentionally keeps the standard `~/.sdkman` home to preserve existing
  Java, Gradle, Kotlin, and other candidates from the previous
  `jetbrains-toolbox` behavior. The installer runs with a temporary `HOME` but
  `SDKMAN_DIR=$HOME/.sdkman`, so SDKMAN's shell init snippets do not get
  appended to shared host dotfiles.
  `java-lsp` installs the latest Eclipse JDTLS snapshot under
  `~/.local/share/dev-toolbox/jdtls`. `kotlin-lsp` installs the latest official
  Kotlin LSP standalone Linux archive under
  `~/.local/share/dev-toolbox/kotlin-lsp`; the installer reads the GitHub
  release tag and constructs the JetBrains CDN URL because the x64 standalone
  archive filename does not include `linux-x64`. GitHub tags may be plain
  `v<version>` or scoped like `kotlin-lsp/v<version>`, so strip any path prefix
  before removing the leading `v`.
- The installer validates tool names before cleanup so a typo cannot remove
  existing selections.
- `/usr/bin/node`, `npm`, `npx`, `uv`, `uvx`, `firecrawl`, `rustup`, `rustc`,
  `cargo`, `bacon`, `rust-analyzer`, `sdk`, `jdtls`, and `kotlin-lsp` are stable symlinks to the
  `dev-toolbox-run` dispatcher. This lets Distrobox export commands before the
  post-setup wizard installs their optional payloads.
- Exports the fourteen tool commands plus `dev-toolbox-install` with `bin:`
  entries.
- Generalizes the shared `.packages` handler to call
  `<app-name-without-box>-install`; this maps both `shell-toolbox` and
  `dev-toolbox` to their matching installer commands without app-specific logic.
- Setup completion text, post-install hints, and app READMEs are emitted by
  `cmd_setup_finish` after wizard selections have been applied. Both direct CLI
  setup and the interactive TUI call this finalizer.

## Pitfalls and notes

- The setup flow exports commands before applying wizard selections. Stable
  system wrappers are therefore required even when a payload is optional.
- Astral's installer checks for commands shadowing the installed uv binaries.
  Its destination is prepended to `PATH` during installation so it finds the
  new payload rather than the stable `/usr/bin/uv` export wrapper.
- Export lookup uses `command -pv`, so wrappers live in `/usr/bin`, not
  `/usr/local/bin` or the shared home.
- Do not install payload executables into `~/.local/bin`; that directory holds
  Distrobox's host wrappers and is shared across all boxes.
- Native npm modules, uv environments, and uv-installed tools target the Ubuntu
  container ABI. Execute them through exported commands or from inside
  `dev-toolbox-box`.
- Rust native crates likewise target the Ubuntu container ABI. JetBrains IDEs
  should be configured to use the exported `cargo`, `rustc`, and `rustup`
  commands rather than toolchains installed by `jetbrains-toolbox`.
- Deselecting Rust must not delete `~/.cargo` or `~/.rustup`; users may already
  have toolchains, cargo caches, and cargo-installed binaries there.
- `rust-lsp` installs `rust-analyzer` as a rustup component into the preserved
  Rust homes. Deselecting `rust-lsp` must not remove the component because that
  would mutate the user's existing Rust install.
- SDKMAN's `sdk` is a shell function, so the exported `sdk` command is a
  dispatcher special case: it sets `SDKMAN_DIR=$HOME/.sdkman`, sources
  `sdkman-init.sh`, then calls `sdk "$@"`. Check readability of the init file,
  not executable bits.
- Deselecting SDKMAN must not delete `~/.sdkman`; users may already have
  important candidates there. Project JDKs installed with
  `sdk install java ...` live under `~/.sdkman/candidates/java/`. JetBrains IDE
  projects can use the `current` symlink there while the IDE process continues
  using JetBrains' bundled JDK.
- The `jdtls` and `kotlin-lsp` dispatch paths prefer
  `~/.sdkman/candidates/java/current` when `JAVA_HOME` is unset, but they
  respect an explicit `JAVA_HOME`. JDTLS needs Java 21 or newer. Kotlin LSP
  currently needs JDK 25.
- Deselecting `java-lsp` or `kotlin-lsp` removes only
  `~/.local/share/dev-toolbox/jdtls` or
  `~/.local/share/dev-toolbox/kotlin-lsp`; it must not touch SDKMAN, Java
  candidates, Rust homes, or editor configuration.
