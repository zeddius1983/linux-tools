## llama-cpp

llama.cpp LLM inference engine with AMD ROCm 7.2.4 GPU acceleration. Provides fast C++ inference for GGUF models with no Python/PyTorch dependency at runtime. Exposes a single `llama` dispatcher command (wrapping upstream `tools.sh`) plus a `llama-serve` convenience server wrapper.

## Implementation

**Base image**: `docker.io/rocm/dev-ubuntu-24.04:7.2.4-complete` for both build and runtime stages.

**Multi-stage build** (required — llama.cpp must be compiled from source):
- Build stage clones `https://github.com/ggml-org/llama.cpp` at `--depth 1` and compiles with HIP.
- CMake flags: `-DGGML_HIP=ON -DGGML_HIP_ROCWMMA_FATTN=ON -DGGML_BACKEND_DL=ON -DGGML_CPU_ALL_VARIANTS=ON`
- Fat arch target: all gfx architectures including gfx1151 (Strix Halo).
- Staged artifacts: `build/bin/*` → `/stage/bin/`, `*.so*` → `/stage/lib/`, `.devops/tools.sh` → `/stage/bin/llama-tools`, `convert_hf_to_gguf.py` → `/stage/bin/`, `gguf-py/` → `/stage/gguf-py/`.

**Runtime layout** — everything lands in `/opt/llama-cpp/`:
- Binaries and `.so` backend plugins co-located (required for `GGML_BACKEND_DL` dynamic loading).
- `llama-tools` (upstream `tools.sh`) lives alongside binaries so its relative paths (`./llama-cli`, `./llama-server`, etc.) resolve correctly.
- `gguf-py/` installed in-place via `pip install --break-system-packages -e`.

**Exported commands** (in `/usr/bin/` for `command -pv` compatibility):
- `llama` — `cd /opt/llama-cpp && exec ./llama-tools "$@"`: dispatcher covering `--run/-r`, `--server/-s`, `--quantize/-q`, `--convert/-c`, `--bench/-b`.
- `llama-serve` — convenience wrapper: sets `HSA_ENABLE_SDMA=0`, binds `0.0.0.0:8080`, calls `llama-server`.

**Model storage**: GGUF files referenced via path; no default location enforced. Distrobox shared `$HOME` means any host path works directly.

**Port**: 8080 (`llama-serve`). Distrobox uses `--network host` so no mapping needed.

## Usage

```bash
# Interactive chat
llama -r -m ~/models/model.gguf

# HTTP API server
llama -s -m ~/models/model.gguf --port 8080
# or using the convenience wrapper
llama-serve -m ~/models/model.gguf

# Quantize a model
llama -q input.gguf output-q4_0.gguf q4_0

# Convert HuggingFace model to GGUF (requires model downloaded locally)
llama -c --outtype f16 ~/models/my-hf-model/

# Benchmark
llama -b -m ~/models/model.gguf
```

## Pitfalls and notes

- **`GGML_BACKEND_DL` co-location**: With `-DGGML_BACKEND_DL=ON`, the runtime searches for backend `.so` files in the same directory as the binary. All `.so` files must be in `/opt/llama-cpp/` alongside the binaries — do not split them.
- **`tools.sh` relative paths**: Upstream `tools.sh` calls `./llama-cli`, `./llama-server`, etc. with relative paths. The `llama` shim does `cd /opt/llama-cpp` first; omitting this will fail with "command not found".
- **`--convert` minimal deps**: Only `gguf-py` is installed. Converting from HuggingFace checkpoints that need `torch`/`transformers` requires installing those inside the container at runtime: `pip install --break-system-packages torch transformers sentencepiece`.
- **`--cap-add SYS_PTRACE` not needed**: Unlike vllm (PyTorch internals), llama.cpp is pure C++ and does not require this capability.
- **Build time**: ~10–20 minutes. Fat arch build compiles ROCm kernels for all listed architectures.
- **ROCm version pinned**: Dockerfile targets `7.2.4-complete` explicitly. To upgrade, change the image tag in both `FROM` lines.
- **`llama-completion` in tools.sh**: Upstream `tools.sh` has a `--run-legacy / -l` branch that calls `./llama-completion`. This binary may not exist in all builds — the flag will error if used.
