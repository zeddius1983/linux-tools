## vllm

OpenAI-compatible LLM inference server with AMD ROCm GPU acceleration. Serves models via HTTP API on port 8000, compatible with any OpenAI client SDK. Packaged using the official `vllm/vllm-openai-rocm` Docker image.

## Implementation

**Base image**: `docker.io/vllm/vllm-openai-rocm:latest`
- Official vLLM image with ROCm + PyTorch pre-installed; no separate vLLM installation needed.
- Also available as `:nightly` for latest development builds.

**Python/vllm path**: `vllm` binary lives at `/usr/local/bin/vllm` in the image. A `/usr/bin/vllm` shim bridges it for `command -pv` (which doesn't search `/usr/local/bin`).

**Server wrapper**: `/usr/bin/vllm-serve` launches `python -m vllm.entrypoints.openai.api_server` with Strix Halo env vars baked in. Accepts all vLLM server flags as arguments (e.g. `--model`, `--max-model-len`, `--dtype`).

**Model storage**: `~/.cache/huggingface` — HuggingFace default, automatically persisted via Distrobox shared `$HOME`. No extra volume mount needed.

**HF_TOKEN**: Not baked into the image. Inherited from the host shell environment at runtime (`export HF_TOKEN=...` in `~/.zshrc`/`~/.bashrc`). Recommended: store via Linux keyring (`secret-tool`) and retrieve in rc files.

**Port**: 8000. Distrobox uses `--network host` so no port mapping needed — `http://localhost:8000` is directly accessible on the host.

**Exports**: `bin:vllm` (full CLI) + `bin:vllm-serve` (convenience server wrapper).

**create_flags**: `--device /dev/kfd --device /dev/dri --group-add 44 --group-add 992 --cap-add SYS_PTRACE --security-opt seccomp=unconfined --ipc=host`
- `/dev/kfd` — AMD ROCm compute device.
- GIDs 44=video, 992=render (numeric because Ubuntu-based images lack these group names in `/etc/group`).
- `--cap-add SYS_PTRACE` — required by vLLM ROCm internals.
- `--ipc=host` — required for ROCm shared-memory IPC.

## Usage

```bash
# Start server with a model
vllm-serve --model Qwen/Qwen3-0.6B

# Full vLLM CLI
vllm serve --model meta-llama/Llama-3-8B-Instruct --max-model-len 8192

# Query the API
curl http://localhost:8000/v1/models
curl http://localhost:8000/v1/completions -H "Content-Type: application/json" \
  -d '{"model": "Qwen/Qwen3-0.6B", "prompt": "Hello", "max_tokens": 100}'
```

## Pitfalls and notes

- **`/usr/local/bin/vllm` path**: `command -pv` (used by `tools.sh` for `bin:` exports) does not search `/usr/local/bin`. The `/usr/bin/vllm` shim is required — confirm the actual binary location after first build with `distrobox enter vllm-box -- which vllm`.
- **Python path uncertainty**: The `vllm-serve` wrapper calls `python -m vllm...`. Confirm `python` resolves correctly inside the container after first build. If the image uses a venv, the wrapper may need `exec /path/to/venv/bin/python` instead.
- **Strix Halo env vars**: Same set as comfyui — `HSA_ENABLE_SDMA=0`, `TORCH_ROCM_AOTRITON_ENABLE_EXPERIMENTAL=1`, `PYTORCH_HIP_ALLOC_CONF=expandable_segments:True`, `TORCH_BLAS_PREFER_HIPBLASLT=1`. May be fixable/removable in future ROCm releases.
- **MIOpen first-run slowness**: First model load triggers GPU kernel compilation. Cache persists at `~/.cache/miopen`.
- **Image size**: `vllm-openai-rocm` images are large (15–25 GB). Allow significant pull/build time.
- **`nightly` tag**: Use `:nightly` for cutting-edge features/fixes; may be less stable than `:latest`.
