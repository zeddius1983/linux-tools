## zsh-box

Host-only installer for Zsh and a curated set of modern shell utilities. Unlike all other apps in this repo, the Distrobox container is used purely as a **portable download/install environment** — nothing runs inside the container at runtime. All binaries end up in `~/.local/bin`, configs in `~/.config/zsh/conf.d/`, and zsh plugins in `~/.local/share/zsh/plugins/`. The container persists so setup can be re-run for updates.

## Implementation

- **Base image**: `ubuntu:24.04` — needs only curl, git, jq, unzip, tar.
- **Export**: `bin:zsh-install` only. The script is `COPY`-ed into the image at `/usr/bin/zsh-install` (not written inline via `RUN printf` because it's ~250 lines).
- **`host-only` marker**: empty `apps/zsh-box/host-only` file. `lib/tui.sh` and `lib/commands.sh` check for this and show dash placeholders plus a final `host` label instead of normal image/box state.
- **Wizard**: `wizard/00-tools.packages` — new `.packages` wizard type added to `lib/wizard.sh`. Handler (`_wizard_apply_packages`) passes selected tool names as a space-separated string to `distrobox enter zsh-box-box -- zsh-install --tools "..."`. Each entry has a 5th `detect` field (bare name → `~/.local/bin/<name>`, `~/` prefix → `$HOME/...`, comma-separated for multiple paths) that `_wizard_run_page` uses to pre-check already-installed tools at runtime.
- **zsh itself**: installed on the host via `distrobox-host-exec sudo apt-get/dnf/pacman install zsh`. OS detected from `/etc/os-release`.
- **Utilities**: static musl binaries downloaded from GitHub releases API via `gh_download_latest()`. Each tool has a hardcoded asset regex pattern for the musl variant.
- **Config strategy**: the managed core config (options, history, completion, keybindings, plugin sourcing, conf.d glob loop, prompt) is written to `~/.config/zsh/zsh-box.zsh` and **fully regenerated on every run** — it carries a `DO NOT EDIT` header. `~/.zshrc` is a thin, **user-owned loader** that just `source`s `zsh-box.zsh` and is **written once, never overwritten** so user customizations below the source line survive re-runs. Per-tool init snippets still go to `~/.config/zsh/conf.d/NN-tool.zsh`, sourced by the glob loop inside `zsh-box.zsh`.
- **`~/.zshrc` handling (3 cases)**: absent → write the loader; exists but has no `zsh/zsh-box.zsh` reference (legacy/foreign rc) → back up `~/.zshrc.bak.TIMESTAMP` and **append** the source line (preserving existing content); already references it → leave untouched. This replaces the old wholesale-overwrite + backup-churn behavior.
- **Plugins**: cloned with `git clone --depth=1` to `~/.local/share/zsh/plugins/`. Sourced directly — no plugin manager.
- **Starship**: downloaded from the latest GitHub release tarball; `gruvbox-rainbow` preset written to `~/.config/starship.toml` only if the file doesn't already exist.
- **delta**: configures `git config --global` for `core.pager`, `delta.side-by-side`, etc. via the `18-delta.zsh` conf.d fragment (runs once on first shell start).
- **zellij completion**: generated at install time (`zellij setup --generate-completion zsh`) and saved as `~/.local/share/zsh/completions/_zellij`. `zsh-box.zsh` prepends that dir to `$fpath` **before** `compinit` so it's autoloaded. There is no `22-zellij.zsh` conf.d fragment anymore (install unconditionally removes any stale copy).

## Pitfalls and notes

- **Never `eval` a tool's `--generate-completion zsh` output**: zellij (and other clap-based tools) emit a `#compdef` script with a **top-level `_arguments` call**. `eval`-ing it at shell startup runs `_arguments` outside a completion context → `_arguments:comparguments: can only be called from completion function`. The earlier `22-zellij.zsh` fragment did exactly this. Fix: write the output to a `_<tool>` file in a `$fpath` dir and let `compinit` autoload it (see `COMP_DIR`). `eval "$(tool init zsh)"` (zoxide/atuin/direnv) is fine — that's shell *init*, not a completion function.
- **`local` outside functions**: bash only allows `local` inside functions. Earlier draft used `local` at top-level script scope — this silently does nothing or errors depending on bash version. All variable declarations in `zsh-install` are plain assignments at script scope.
- **`gh_download_latest` regex**: the jq `test()` regex must be anchored (`^...$`) to avoid false-positive matches between e.g. `bat-v0.24.0-x86_64-unknown-linux-musl.tar.gz` and the `-gnu` variant.
- **fzf keybindings**: the key-bindings and completion shell scripts are separate from the binary and must be downloaded from the source repo at the same tag. They are saved to `~/.local/share/` and sourced in `10-fzf.zsh`.
- **atuin / zoxide / starship downloads**: use GitHub release tarballs via `gh_download_latest()` instead of `curl | bash` install scripts, which avoids executing mutable remote shell scripts during setup.
- **yazi zip layout**: the zip extracts to `yazi-<triple>/yazi` (and optionally `ya`). The full subdir path must be used when calling `install`.
- **`host-only` TUI column width**: `app_status_detail()` prints 7- and 32-column dash placeholders plus a final `host` label so the host-only row stays aligned with the normal `img_id | img_ref | box_str` layout.
- **detect field binary name mismatches**: `ripgrep` installs as `rg`, `tealdeer` as `tldr`, zsh plugins are directories at `~/.local/share/zsh/plugins/<name>`. The detect field in `00-tools.packages` must use the actual installed name, not the tool name. `atuin` uses a comma-separated detect field to cover both `~/.local/bin/atuin` (new) and `~/.atuin/bin/atuin` (old installs).
- **shell change**: `chsh` requires the shell to be listed in `/etc/shells`. Installing via the host package manager handles this automatically. Static zsh builds do not register in `/etc/shells`.
- **`$COMP_DIR` must not be group/other-writable**: it's prepended to `$fpath`, and `compinit`'s security audit (`compaudit`, run strictly — no `-u`/`-i`) rejects group- or world-writable fpath dirs. A umask of 002 (Ubuntu per-user groups) makes `mkdir` create it `775`, so the install `chmod g-w,o-w "$COMP_DIR"` right after creating it. Symptom when this regresses: entering *any* box prints `compinit: insecure directories … abort? [y]` followed by `compdef: command not found`. The trigger is subtle — compinit caches its audit in the zcompdump and only re-audits when the cache is invalidated, e.g. when another container adds a new fpath entry (the claude-code box gained `/usr/local/share/zsh/site-functions` when Node.js was bundled for npx). So a long-latent 775 dir can suddenly start failing after an unrelated container changes its fpath.
