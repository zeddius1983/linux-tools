## jetbrains-toolbox

JetBrains Toolbox App packaged in a Distrobox container. Toolbox manages
JetBrains IDEs (IDEA, PyCharm, GoLand, etc.) installed into the shared `$HOME`
— so IDE binaries and their `.desktop` files appear on the host immediately
without any extra export step.

## Implementation

- Base: `docker.io/ubuntu:24.04` (officially supported by JetBrains; 22.04 and 24.04 LTS listed)
- Export: `gui:jetbrains-toolbox:JetBrains Toolbox` — system-tray app, `Terminal=false`
- Toolbox binary downloaded at image-build time via JetBrains release API (`code=TBA`) to `/opt/jetbrains-toolbox/`; symlinked to `/usr/bin/jetbrains-toolbox`
- The image intentionally keeps only GUI/runtime dependencies needed to launch
  Toolbox. Language runtimes, compilers, and project CLIs belong in
  `dev-toolbox`.
- The wrapper only refreshes Toolbox tray icons in shared `$HOME` before
  launching `/opt/jetbrains-toolbox/bin/jetbrains-toolbox`.

On first run, Toolbox creates `~/.local/share/applications/jetbrains-toolbox.desktop` in the shared `$HOME`, overwriting the distrobox-exported one. This is fine — the Toolbox-managed `.desktop` is better.

IDEs installed by Toolbox go to `~/.local/share/JetBrains/Toolbox/apps/`
(shared `$HOME`) and are launched from the host directly. JetBrains bundles its
own JDK for the IDE process; project toolchains should be configured from
`dev-toolbox` exports.

## Pitfalls and notes

- **No language toolchain side effects**: launching Toolbox must not install
  Rust, SDKMAN, Java, or general development tools. Rust is provided by
  `dev-toolbox`; SDKMAN/Java is also provided there through an exported `sdk`
  wrapper because `sdk` is a shell function, not a normal binary. The SDKMAN
  home remains `~/.sdkman` so existing candidates are preserved. Rust homes
  remain `~/.cargo` and `~/.rustup` so existing toolchains and cargo-installed
  tools are preserved.
- **Toolbox `.desktop` overwrite**: Toolbox replaces the distrobox-exported `.desktop` on first run. Expected behaviour — do not re-export after initial setup unless Toolbox is reinstalled.
- **Tray icon shows grey exclamation mark (recurring)**: Toolbox registers its AppIndicator with `IconThemePath=/opt/jetbrains-toolbox/bin/` — a container-only path the host Cinnamon panel cannot read. Cinnamon falls back to looking up the icon name `toolbox-tray-color` in the system theme. Fix: wrapper unconditionally copies the PNG as both `jetbrains-toolbox.png` and `toolbox-tray-color.png` into `~/.local/share/icons/hicolor/256x256/apps/`, and aliases JetBrains' own scalable SVG (`jetbrains-toolbox.svg`) as `toolbox-tray-color.svg` in `hicolor/scalable/apps/` (when it exists — Toolbox creates it on first run). Then runs `gtk-update-icon-cache -f -t ~/.local/share/icons/hicolor`. The SVG alias is critical for a sharp icon — the PNG is a blurry fallback for the very first launch only. **Do not guard copies with `[[ ! -f ]]`** — stale icon recurred after rebuilds because the old file blocked the copy. After any cache or rebuild change, flush Cinnamon's in-memory icon cache with `cinnamon --replace &` on the host.
- **IDEs outside the container**: Toolbox-generated `.desktop` files for individual IDEs invoke the IDE binary directly on the host (from `$HOME`). They run outside the container environment. This is fine because JetBrains IDEs ship a bundled JDK and most UI libs are present on any modern desktop.
